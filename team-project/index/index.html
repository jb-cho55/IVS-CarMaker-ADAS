시스템 아키텍처

설계 배경

기존 구조는 역할 단위(횡제어/종제어/모드/경로/궤적)로 모듈을 나눠 한 시나리오를 처리하려면 여러 Lib를 동시에 건드려야 했음. 시나리오 단위로 재편하여 각 모듈의 책임을 명확히 함.

  • 주행 시나리오 = 일반 주행 + 추월 (Lib_Driving)
  • 톨게이트 시나리오 = 하이패스 차선 진입 + 통과 (Lib_Tollgate)
  • 주차 시나리오 = A* 경로 + 주차 maneuver (Lib_Parking, 전·후진 모두)

스케줄링 / 트리거

Stateflow chart 기반 (수정 금지)

Stateflow chart (내부):
  i = 0;
  i = i + 1;                          // 매 base step마다
  if (mod(i,10)  == 0) do_Logic_10ms();
  if (mod(i,50)  == 0) do_Logic_50ms();
  if (mod(i,100) == 0) do_Logic_100ms();

chart는 function-call 이벤트 3종을 발행. 각 이벤트는 chart 옆의 Function-Call Split을 거쳐 2개 채널(_1, _2)로 분기되어 Goto로 broadcast.

Lib별 Trigger 직결 (chart 출력 직접 구독)

Lib주기From 태그
Lib_MissionManager10msdo_Logic_10ms_1
Lib_Driving10msdo_Logic_10ms_1
Lib_VehicleController_Driving10msdo_Logic_10ms_2
Lib_Tollgate100msdo_Logic_100ms_1
Lib_Parking50msdo_Logic_50ms_1
Lib_VehicleController_Parking50msdo_Logic_50ms_2

⚠ MissionManager + Driving은 동일 채널(do_Logic_10ms_1)을 공유. Goto/From은 fan-out 가능하므로 각각 별도 From 블록을 사용하면 OK.

미사용 예약 슬롯

  • do_Logic_100ms_2 — 100ms 추가 task용
  • 일부 50ms 채널 (모두 사용 중)

새 task 추가 시 위 슬롯을 사용. chart는 절대 수정하지 않음.

왜 chart 기반인가

  • Deterministic execution: CarMaker simstep 안에서 항상 같은 순서로 실행
  • Multi-rate without rate transition: 모든 task가 같은 base clock 위에서 동기 실행
  • 명시적 스케줄링: 차트만 보면 누가 언제 실행되는지 한눈에 파악
  • CarMaker 템플릿 호환: generic_IVS.mdl 표준
