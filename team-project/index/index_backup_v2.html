 Werling Frenet 수식 (참고용)

%% Quintic polynomial (횡)
d(t) = a0 + a1·t + a2·t² + a3·t³ + a4·t⁴ + a5·t⁵
%% 6개 계수, 6개 boundary condition으로 결정

%% Quartic polynomial (종)
s(t) = b0 + b1·t + b2·t² + b3·t³ + b4·t⁴
%% 5개 계수, 5개 boundary condition

%% 비용
c_lat = K_J·∫d⃛²dt + K_T·T + K_D·(d_f - d_target)²
c_lon = K_J·∫s⃛²dt + K_T·T + K_V·(s_f_dot - v_target)²
c_tot = K_LAT·c_lat + K_LON·c_lon + K_TTC/ttc

%% 제약 (모두 만족해야 후보 채택)
max|v(t)|  ≤ V_MAX_FRENET (30 m/s)
max|a(t)|  ≤ ACC_MAX_FRENET (4 m/s²)
max|κ(t)| ≤ K_MAX_CURV (0.20 1/m)

⚠️ 주의 + 참고

  • • cte_offset = cte_in − target_offset 부호 주의 — 다음 turn에 검증
  • • Lateral 컨트롤러(Dev-E)가 받는 입력 신호이므로 부드러움이 중요 (jerk 최소화)

  • • 04 patch의 상수: K_J=0.1, K_T=1.0, K_D=1.0, K_V=1.0, K_LAT=1.0, K_LON=1.0

            lane_cost(i) = 9999;  %% 멀리 있는 차선은 거의 무한 비용
        end
    end
    [~, idx] = min(lane_cost);
    target_lane = int32(idx);
  3. Step 3: 옆차선 안전 hard constraint 
    %% 옆 차선에 트래픽 너무 가까우면 그 차선 막음
    if abs(lat_dist - lane_offset(i)) < 1.5 && abs(long_dist) < 10

function acc_cmd = longitudinal_controller_fcn(v_ego, v_target, front_gap, mode)
%#codegen
% LONGITUDINAL_CONTROLLER_FCN — 종방향 제어 (ACC + LK)
% Owner: Dev-A
%
% Inputs:
%   v_ego     : 자차 속도 [m/s]
%   v_target  : 목표 속도 [m/s]
%   front_gap : 선행차 거리 [m] (없으면 999)
%   mode      : int32, mission mode
% Output:
%   acc_cmd   : 목표 가속도 [m/s²], 범위 [-5, +3]

persistent v_int
if isempty(v_int), v_int = 0; end

% --- Dev-A: 여기에 PID + ACC 로직 ---
safe_dist = 10 + 2*v_ego;
e_d = front_gap - safe_dist;
e_v = v_target - v_ego;
e_eff = min(e_d, e_v);

Kp = 0.5; Ki = 0.1;
v_int = max(min(v_int + e_eff*0.001, 2.0), -2.0);
acc_cmd = Kp*e_eff + Ki*v_int;

% 비상 제동
if front_gap < 5, acc_cmd = -5; end

% 한계
acc_cmd = max(min(acc_cmd, 3.0), -5.0);
end

 알고리즘 설명

1) 안전 거리 모델: safe_dist = 10 + 2·v_ego — 2초 룰 (속도 비례).

2) 두 오차 계산:

  • e_d = front_gap − safe_dist (ACC 추종 오차)
  • e_v = v_target − v_ego (속도 추종 오차)
  • e_eff = min(e_d, e_v) ← 더 보수적인 쪽 선택

3) PI 제어 + Anti-windup:

  • 적분기 saturation ±2.0 (windup 방지)
  • acc_cmd = Kp·e_eff + Ki·v_int (Kp=0.5, Ki=0.1)

4) 안전 한계: front_gap < 5m → 비상 제동 -5 m/s², 최종 출력 clip [-5, +3]

function steer_cmd = lateral_controller_fcn(cte, mode)
%#codegen
% LATERAL_CONTROLLER_FCN — 횡방향 제어 (PID 등)
% Owner: Dev-E
%
% Inputs:
%   cte  : Cross-Track Error [m] (DevDist or modified by trajectory)
%   mode : int32, 1=LK, 3=LC, 5=Parking, etc.
% Output:
%   steer_cmd : 앞바퀴 조향각 [rad], 범위 ±pi/6

persistent e_prev e_int
if isempty(e_prev), e_prev = 0; e_int = 0; end

% --- Dev-E: 여기에 PID/Stanley/Pure Pursuit 구현 ---
Kp = 1.5; Ki = 0.3; Kd = 0.4;
e  = -cte;
de = e - e_prev;
e_int = max(min(e_int + e*0.001, 3.0), -3.0);
e_prev = e;
steer_cmd = Kp*e + Ki*e_int + Kd*de;

% 조향각 한계
steer_cmd = max(min(steer_cmd, pi/6), -pi/6);
end

 알고리즘 설명

1) 오차 부호 규칙: e = −cte

  • DevDist > 0 (차로 오른쪽으로 벗어남) → e < 0 → 좌조향 (+)
  • CarMaker 컨벤션과 일치하도록 부호 반전 처리

2) PID 게인 (Day6 검증)

  • Kp=1.5, Ki=0.3, Kd=0.4
  • 적분기 saturation ±3.0 (anti-windup)

3) 조향각 한계: ±π/6 rad (±30°) — 차량 물리 한계

4) Persistent state: 이전 오차 (Kd용), 적분 누적 (Ki용)

function mode = mode_manager_fcn(ego_v, ego_x, ego_y, ego_yaw, traffic_x, traffic_y, target_lane, current_lane)
%#codegen
% MODE_MANAGER_FCN — 미션 모드 결정 (LK/OT/LC/TG/Parking 등)
% Owner: Dev-B
%
% Outputs:
%   mode : int32
%       0 = IDLE
%       1 = LANE_KEEP (default)
%       2 = OVERTAKE (decision in progress)
%       3 = LC_OUT  (lane change out)
%       4 = LC_PASS (passing in adjacent lane)
%       5 = LC_BACK (returning to original lane)
%       6 = TOLLGATE
%       7 = PARKING
%       9 = EMERGENCY

persistent state t_in_state
if isempty(state), state = int32(1); t_in_state = 0; end

TS = 0.001;
TRIGGER_DIST = max(40.0, ego_v * 3.0);
PASS_DIST = -15.0;

% 트래픽 종/횡 거리
dx = traffic_x - ego_x;
dy = traffic_y - ego_y;
long_dist = cos(ego_yaw)*dx + sin(ego_yaw)*dy;
lat_dist  = -sin(ego_yaw)*dx + cos(ego_yaw)*dy;

t_in_state = t_in_state + TS;

% --- Dev-B: 여기에 추월/모드 결정 로직 ---
switch state
    case int32(1)  % LK
        if long_dist > 0.5 && long_dist < TRIGGER_DIST && abs(lat_dist) < 1.8 && ego_v > 5
            state = int32(3); t_in_state = 0;
        end
    case int32(3)  % LC_OUT
        if t_in_state >= 4.0
            state = int32(4); t_in_state = 0;
        end
    case int32(4)  % LC_PASS
        if long_dist < PASS_DIST || long_dist > 150
            state = int32(5); t_in_state = 0;
        end
    case int32(5)  % LC_BACK
        if t_in_state >= 4.0
            state = int32(1); t_in_state = 0;
        end
end

mode = state;
end

 알고리즘 설명 (State Machine)

1) Frenet 좌표 변환: 트래픽을 자차 기준 종/횡 거리로

long_dist = cos(yaw)·dx + sin(yaw)·dy
lat_dist  = −sin(yaw)·dx + cos(yaw)·dy

2) 동적 트리거 거리: TRIGGER_DIST = max(40, v_ego·3) (속도 비례, 최소 40m)

3) 상태 전이:

  • LK → LC_OUT: 선행차 가까움 (long_dist < trigger) + 같은 차선 (|lat_dist| < 1.8m) + 속도 > 5
  • LC_OUT → LC_PASS: 4초 경과 (차선 변경 완료)
  • LC_PASS → LC_BACK: 추월 완료 (long_dist < −15) 또는 신호 사라짐 (> 150)
  • LC_BACK → LK: 4초 경과 (원 차선 복귀)

4) 시간 누적: t_in_state persistent — TS = 0.001s씩 증가

function cte_offset = trajectory_planning_fcn(cte_in, mode, ego_x, ego_y, ego_yaw, ego_v)
%#codegen
% TRAJECTORY_PLANNING_FCN — LC/LK 궤적 생성 (Werling Frenet 등)
% Owner: Dev-C
%
% Inputs:
%   cte_in : 원본 Cross-Track Error [m] (DevDist)
%   mode   : int32, 1=LK, 3=LC_OUT, 4=LC_PASS, 5=LC_BACK
%   ego_*  : 자차 상태 (Werling planner에 사용)
% Output:
%   cte_offset : 보정된 CTE (lateral controller 입력)

persistent state t_in_state target_offset
if isempty(state)
    state = int32(0); t_in_state = 0; target_offset = 0;
end

TS = 0.001;
LANE_WIDTH = 3.0;
LC_TIME = 4.0;

% --- Dev-C: 여기에 Werling Frenet 다중 후보 trajectory 구현 ---
% 현재는 단순 quintic ramp (placeholder)

if mode == int32(3)  % LC_OUT
    t_in_state = t_in_state + TS;
    s = min(1, t_in_state / LC_TIME);
    target_offset = -LANE_WIDTH * (10*s^3 - 15*s^4 + 6*s^5);
elseif mode == int32(4)  % LC_PASS
    target_offset = -LANE_WIDTH;
    t_in_state = 0;
elseif mode == int32(5)  % LC_BACK
    t_in_state = t_in_state + TS;
    s = min(1, t_in_state / LC_TIME);
    target_offset = -LANE_WIDTH * (1 - (10*s^3 - 15*s^4 + 6*s^5));
else  % LK
    target_offset = 0;
    t_in_state = 0;
end

cte_offset = cte_in - target_offset;
end

 알고리즘 (현재 placeholder → 개선 방향)

현재: Quintic Offset Ramp

s(t) = 10t³ − 15t⁴ + 6t⁵     %% C² 연속 ramp (0→1)
offset(t) = −LANE_WIDTH · s(t/LC_TIME)
cte_offset = cte_in − target_offset

v2 권장: Werling Frenet 다중 후보 (DEV_C patch 참고)

  • 횡: Quintic polynomial (양끝 위치+속도+가속도 고정)
  • 종: Quartic polynomial (final position 자유)
  • Nd × Nt 후보 (18개) → 비용 최소 선택
  • 제약: |v|≤30 m/s, |a|≤4 m/s², |κ|≤0.20 1/m
  • 실시간: ~1.2 ms (100ms 주기 대비 1.2%)

function [target_lane, lane_cost] = path_planning_fcn(ego_x, ego_y, ego_yaw, traffic_x, traffic_y, current_lane)
%#codegen
% PATH_PLANNING_FCN — 경로 계획 + Lane Risk 계산
% Owner: Dev-D
%
% Inputs:
%   ego_*       : 자차 상태
%   traffic_x,y : 가장 위협적인 트래픽 (또는 array)
%   current_lane: int32 (1~MAX_LANE)
% Outputs:
%   target_lane : int32, 목표 차선
%   lane_cost   : double [4x1], 차선별 비용 (낮을수록 좋음)

MAX_LANE = int32(4);

% --- Dev-D: 여기에 LaneRisk + Frenet 경로 계획 ---
% 현재는 단순 placeholder

lane_cost = zeros(4, 1);

% 트래픽 종/횡 거리 계산
dx = traffic_x - ego_x;
dy = traffic_y - ego_y;
long_dist = cos(ego_yaw)*dx + sin(ego_yaw)*dy;
lat_dist  = -sin(ego_yaw)*dx + cos(ego_yaw)*dy;

% 단순 비용: 현재 차선에 가까울수록 비용 ↓
for i = 1:4
    lane_cost(i) = abs(double(i) - double(current_lane)) * 5.0;
end

% 선행차 가까우면 현재 차선 비용 ↑
if long_dist > 0.5 && long_dist < 50 && abs(lat_dist) < 1.8
    lane_cost(current_lane) = lane_cost(current_lane) + 100;
end

% 최저 비용 차선 선택
[~, idx] = min(lane_cost);
target_lane = int32(idx);
end

 알고리즘 설명

1) 현재 → 단순 placeholder:

  • 차선 비용 = 현재 차선과의 거리 × 5
  • 선행차 (가까움+같은 차선) → 현재 차선에 +100 패널티
  • [~, idx] = min(lane_cost) → target_lane

2) v2 권장 — 상세 Lane Cost:

lane_cost(i) = K_BLOCK · blocked(i)
             + K_TTC / max(ttc(i), 0.5)
             + K_GAP / max(front_gap(i), 5.0)
             + K_LATBIAS · |i − current_lane|
  • K_BLOCK=1000 (안전 hard constraint)
  • K_TTC=50, K_GAP=30, K_LATBIAS=5 (soft optimization)
  • 인접 차선만 고려 (current ± 1)

함정 1

시간 값은 감소하지 않아야 합니다

증상

From Workspace에서 workspace 입력으로 유효하지 않은 작업 공간 변수
시간 값은 감소하지 않아야 합니다.

원인

base workspace의 waypoints가 [x, y] 877×2 형식. From Workspace 블록은 첫 column을 시간으로 해석. 도로 곡선부에서 x좌표가 감소하므로 시간 감소로 인식.

해결

setup_paths.m이 자동으로 [time, x, y] 877×3 형식으로 변환. From Workspace 블록은 Do Not Modify 영역이라 그대로 둠.

함정 2

BusSelector 입력이 Bus 신호가 아닙니다

증상

Bus Selector 블록에 대한 입력은 가상 버스 신호 또는
비가상 버스 신호여야 합니다.

원인

BusCreator3 → Goto[Ego_Global_Pos] → From → BusSelector 경로에서 Bus type 정보 손실.

해결

BusSelector 제거. Modify 영역에 신규 Read CM Dict 3개 추가:

  • Read_Ego_X_Lib → Goto[Ego_Global_X]
  • Read_Ego_Y_Lib → Goto[Ego_Global_Y]
  • Read_Ego_Yaw_Lib → Goto[Ego_Yaw]
함정 3

Goto/From이 atomic 경계 못 넘음

증상

Goto/From connections cannot cross nonvirtual subsystem boundaries.

원인

Lib_*는 Function-Call Triggered Subsystem = 강제 nonvirtual atomic. 안과 밖 신호는 Inport/Outport로만 전달 가능.

해결

Lib 안의 From/Constant 모두 제거. Lib에 Inport/Outport 추가. 외부에서 From/Constant → Lib.Inport, Lib.Outport → Goto.

Lib 안:  Inport_1 → Wrapper → Outport
Lib 밖:  From/Constant → Lib.Inport,  Lib.Outport → Goto
함정 4

function-call signal 단순 분기 불가

증상

function-call signal is branched incorrectly.
needs to be recreated using a Function-Call Split block.

원인

같은 트리거(do_Logic_10ms_1)를 2개 Lib(Long, Lat)가 단순 line branch로 받음.

해결

FCS_10ms_LongLat Function-Call Split 블록 추가 (1 in → 2 out).

함정 5

Day6 시나리오 활성화로 변수 못 찾음

증상

Variable 'Traffic.T01.rzv' not in dictionary

day7_final TestRun에는 Traffic.T22~T28만 있는데, Day6/Scenario 1 서브시스템이 동시에 활성이라 Traffic.T01.rzv 변수를 찾으려 함.
